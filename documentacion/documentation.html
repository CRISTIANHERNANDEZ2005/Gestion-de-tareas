Documentación Técnica del Proyecto

¡Hola! Si estás leyendo esta documentación, probablemente eres un desarrollador que quiere entender cómo construí este Gestor de Tareas. Te explicaré cómo desarrollé cada parte del proyecto, desde la arquitectura hasta los detalles de implementación, para que puedas mantenerlo o extenderlo fácilmente.

Backend Robusto

Desarrollé una API RESTful con Python Flask y PostgreSQL (Neon) para gestionar eficientemente los datos de usuarios y tareas.

Autenticación Segura

Implementé autenticación JWT para proteger las rutas y mantener sesiones seguras.

Base de Datos Moderna

Utilicé Neon como servicio de base de datos PostgreSQL serverless para escalabilidad y rendimiento.

Mi Enfoque de Desarrollo

Cuando desarrollé este proyecto, me aseguré de seguir buenas prácticas de programación, manteniendo un código limpio, bien documentado y fácil de entender. Mi objetivo fue crear una aplicación funcional pero también escalable para futuras mejoras.

Arquitectura del Sistema

Para este proyecto, elegí una arquitectura cliente-servidor tradicional con una clara separación entre frontend y backend. Esta decisión me permitió mantener el código organizado y facilitar el mantenimiento futuro.

Frontend (Cliente)

En el frontend, decidí utilizar tecnologías web estándar pero con un enfoque moderno:

  • HTML5: Estructuré la interfaz con etiquetas semánticas para mejorar la accesibilidad y SEO.
  • CSS/: Creé estilos personalizados con un enfoque en la experiencia de usuario y el diseño responsivo.
  • JavaScript (ES6+): Implementé interactividad y comunicación con el backend usando las últimas características de JavaScript.
  • Módulos ES6: Organicé el código en módulos reutilizables para mantener una estructura limpia y evitar el código espagueti.

Backend (Servidor)

Para el backend, opté por tecnologías robustas y bien establecidas:

  • Python Flask: Desarrollé la lógica del servidor y la API RESTful aprovechando la simplicidad y potencia de Flask.
  • PostgreSQL (Neon): Elegí Neon como servicio de base de datos PostgreSQL serverless por su confiabilidad y escalabilidad.
  • SQLAlchemy ORM: Utilicé SQLAlchemy para acceder a la base de datos de forma segura y prevenir inyecciones SQL.
  • JWT: Implementé JSON Web Tokens para autenticación segura entre frontend y backend.

Base de Datos que Diseñé

Para la base de datos, creé una estructura eficiente que cumple con todos los requisitos del proyecto. Utilicé Neon como servicio de base de datos PostgreSQL serverless para mejor escalabilidad y rendimiento:

Modelos de Datos

Definí dos modelos principales usando SQLAlchemy ORM:

Modelo Usuario

class Usuario(db.Model):
    __tablename__ = 'usuarios'
    
    id = db.Column(db.Integer, primary_key=True)
    identificacion = db.Column(db.String(20), unique=True, nullable=False)
    nombre = db.Column(db.String(100), nullable=False)
    apellido = db.Column(db.String(100), nullable=False)
    contrasena = db.Column(db.String(255), nullable=False)
    creado_en = db.Column(db.DateTime, default=datetime.utcnow)
    actualizado_en = db.Column(db.DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
    
    # Relación con las tareas del usuario
    tareas = db.relationship('Tarea', backref='usuario', lazy=True, cascade="all, delete-orphan")

Modelo Tarea

class Tarea(db.Model):
    __tablename__ = 'tareas'
    
    id = db.Column(db.Integer, primary_key=True)
    usuario_id = db.Column(db.Integer, db.ForeignKey('usuarios.id'), nullable=False)
    titulo = db.Column(db.String(100), nullable=False)
    descripcion = db.Column(db.Text)
    fecha_limite = db.Column(db.Date)
    creado_en = db.Column(db.DateTime, default=datetime.utcnow)
    actualizado_en = db.Column(db.DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
    
    # Restricción única: un usuario no puede tener dos tareas con el mismo título
    __table_args__ = (
        db.UniqueConstraint('usuario_id', 'titulo', name='unique_titulo_usuario'),
    )
Tabla Campos Relaciones
usuarios id, identificacion, nombre, apellido, contrasena, creado_en, actualizado_en Tiene muchas tareas (one-to-many)
tareas id, usuario_id, titulo, descripcion, fecha_limite, creado_en, actualizado_en Pertenece a un usuario (many-to-one)

Decisiones de Diseño

Cuando diseñé esta estructura, tomé varias decisiones importantes:

  • FOREIGN KEY: Usé una clave foránea en la tabla tareas para vincular cada tarea a un usuario específico.
  • UNIQUE CONSTRAINT: Implementé una restricción única para que un usuario no pueda tener dos tareas con el mismo título.
  • TIMESTAMPS: Agregué campos de creación y actualización automáticos para llevar registro de cuándo se crearon y modificaron los registros.
  • CASCADE DELETE: Configuré eliminación en cascada para que cuando un usuario se elimine, también se eliminen sus tareas.

Script SQL de la Base de Datos

A continuación se presenta el script SQL utilizado para crear la base de datos y las tablas:

-- Creación de la base de datos para el gestor de tareas
CREATE DATABASE gestor_tareas_db;

-- Creación de la tabla usuarios
CREATE TABLE IF NOT EXISTS usuarios (
    id SERIAL PRIMARY KEY,
    identificacion VARCHAR(20) UNIQUE NOT NULL,
    nombre VARCHAR(100) NOT NULL,
    apellido VARCHAR(100) NOT NULL,
    contrasena VARCHAR(255) NOT NULL,
    creado_en TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    actualizado_en TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Creación de índice para identificacion
CREATE INDEX IF NOT EXISTS idx_identificacion ON usuarios(identificacion);

-- Creación de la tabla tareas con relación a usuarios
CREATE TABLE IF NOT EXISTS tareas (
    id SERIAL PRIMARY KEY,
    usuario_id INTEGER NOT NULL,
    titulo VARCHAR(100) NOT NULL,
    descripcion TEXT,
    fecha_limite DATE,
    creado_en TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    actualizado_en TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (usuario_id) REFERENCES usuarios(id) ON DELETE CASCADE,
    UNIQUE (usuario_id, titulo)
);

-- Creación de índice para usuario_id
CREATE INDEX IF NOT EXISTS idx_usuario_id ON tareas(usuario_id);

-- Creación de función y trigger para actualizar automáticamente actualizado_en
CREATE OR REPLACE FUNCTION actualizar_timestamp()
RETURNS TRIGGER AS $$
BEGIN
    NEW.actualizado_en = CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Triggers para actualizar automáticamente actualizado_en
CREATE TRIGGER trigger_actualizar_usuarios
    BEFORE UPDATE ON usuarios
    FOR EACH ROW
    EXECUTE FUNCTION actualizar_timestamp();

CREATE TRIGGER trigger_actualizar_tareas
    BEFORE UPDATE ON tareas
    FOR EACH ROW
    EXECUTE FUNCTION actualizar_timestamp();

Crea y activa un entorno virtual para aislar las dependencias:

# En macOS/Linux
python3 -m venv venv
source venv/bin/activate

3 Instalar Dependencias

Instala las dependencias del proyecto usando pip:

pip install -r requirements.txt

4 Configurar Variables de Entorno

Crea un archivo .env basado en el .env.example con tu configuración de Neon:

DATABASE_URL=postgresql://usuario:contraseña@host/neondb?sslmode=require
JWT_SECRET_KEY=tu_clave_secreta_segura
FLASK_ENV=development

5 Ejecutar Migraciones

Ejecuta las migraciones para crear las tablas en la base de datos:

flask db upgrade

6 Iniciar la Aplicación

Finalmente, inicia el servidor de desarrollo:

python run.py

La aplicación estará disponible en http://localhost:5000

Requerimientos Funcionales y No Funcionales

A continuación se detallan los Requerimientos funcionales y no funcionales que guían el desarrollo y funcionamiento de esta aplicación:

Requerimientos Funcionales

Los Requerimientos funcionales definen qué debe hacer el sistema en términos de sus funcionalidades y características:

Gestión de Usuarios

  • RF-001: El sistema debe permitir el registro de nuevos usuarios con identificación única, nombre, apellido y contraseña.
  • RF-002: El sistema debe permitir el inicio de sesión de usuarios registrados mediante identificación y contraseña.
  • RF-003: El sistema debe permitir a los usuarios actualizar su información personal y contraseña.
  • RF-004: El sistema debe permitir el cierre de sesión seguro de los usuarios.

Gestión de Tareas

  • RF-005: El sistema debe permitir a los usuarios crear nuevas tareas con título, descripción y fecha límite.
  • RF-006: El sistema debe mostrar a los usuarios una lista de todas sus tareas.
  • RF-007: El sistema debe permitir a los usuarios editar sus tareas existentes.
  • RF-008: El sistema debe permitir a los usuarios eliminar sus tareas.
  • RF-009: El sistema no debe permitir que un usuario tenga dos tareas con el mismo título.

Validaciones y Reglas de Negocio

  • RF-010: El sistema debe validar que la identificación del usuario tenga al menos 8 dígitos numéricos.
  • RF-011: El sistema debe validar que las contraseñas tengan al menos 6 caracteres.
  • RF-012: El sistema debe validar que los títulos de las tareas tengan entre 5 y 100 caracteres.
  • RF-013: El sistema debe validar que las descripciones de las tareas tengan al menos 10 caracteres.
  • RF-014: El sistema debe validar que las fechas límite de las tareas sean futuras.

Requerimientos No Funcionales

Los Requerimientos no funcionales definen cómo debe comportarse el sistema en términos de calidad, rendimiento y restricciones:

Rendimiento

  • RNF-001: El sistema debe responder a las solicitudes del usuario en menos de 2 segundos bajo carga normal.
  • RNF-002: El sistema debe ser capaz de manejar al menos 100 usuarios concurrentes.
  • RNF-003: El sistema debe mantener tiempos de respuesta de base de datos menores a 500ms para operaciones comunes.

Seguridad

  • RNF-004: El sistema debe almacenar las contraseñas de los usuarios utilizando hashing seguro (Werkzeug).
  • RNF-005: El sistema debe proteger las rutas mediante autenticación JWT.
  • RNF-006: El sistema debe expirar los tokens JWT después de 1 hora de inactividad.
  • RNF-007: El sistema debe utilizar conexiones SSL/TLS para comunicarse con la base de datos.

Disponibilidad y Confiabilidad

  • RNF-008: El sistema debe tener una disponibilidad del 99.5% durante el horario comercial.
  • RNF-009: El sistema debe manejar errores de manera elegante y proporcionar mensajes de error útiles.
  • RNF-010: El sistema debe realizar rollback automático en caso de errores de base de datos.

Usabilidad

  • RNF-011: El sistema debe ser completamente responsivo y funcionar en dispositivos móviles y de escritorio.
  • RNF-012: El sistema debe proporcionar retroalimentación visual inmediata para todas las acciones del usuario.
  • RNF-013: El sistema debe presentar mensajes de error comprensibles en español.

Mantenibilidad

  • RNF-014: El sistema debe seguir una arquitectura modular basada en blueprints de Flask.
  • RNF-015: El sistema debe utilizar migraciones de base de datos para gestionar cambios en el esquema.
  • RNF-016: El código debe estar completamente documentado con docstrings en español.

Priorización de Requisitos

Los Requerimientos se han priorizado según su importancia para el funcionamiento básico del sistema:

Alta Prioridad

Requerimientos esenciales para el funcionamiento básico del sistema (RF-001 al RF-004, RNF-004, RNF-005)

Media Prioridad

Requerimientos importantes para una experiencia completa (RF-005 al RF-014, RNF-001, RNF-006 al RNF-013)

Baja Prioridad

Requerimientos deseables pero no críticos (RNF-002, RNF-003, RNF-014 al RNF-016)
